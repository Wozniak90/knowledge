Proč Clean Core vznikl a co znamená v praxi

Clean Core není „zákaz customizací“ ani ideologie „žádné Z objekty“. SAP jej definuje jako sadu řídicích principů, které mají podporovat průběžnou (kontinuální) transformaci a modernizaci ERP tím, že udrží digitální jádro agilní, inovativní a efektivní. V praxi to znamená systematicky zamezit tomu, aby se S/4HANA jádro v čase „zatížilo“ obtížně udržitelným kódem, nedokumentovanými zásahy a křehkými integracemi, které zpomalují upgrady a zvyšují TCO.[1][2][3]

Historický důvod je jednoduchý: tradiční on‑premise ERP (typicky s ECC‑mindsetem) byl často rozšiřován „nejkratší cestou“ přímo v jádře – implicitními enhancementy, přímými přístupy na tabulky, Z funkčními moduly volajícími interní objekty, modifikacemi standardu, atd. Takový přístup je rychlý v krátkém horizontu, ale dlouhodobě vytváří technický dluh a opakované náklady při každé transformaci (S/4 konverze, FPS, upgrady, retrofit, regresní testování).[4]

Ve světě S/4HANA se navíc mění „operační rytmus“ a očekávání: cloudové edice směřují k častějším inovacím a menším „big‑bang“ upgrade projektům. Clean Core je proto navržen tak, aby:

  • držel jádro co nejblíže standardu,
  • přesunul diferenciaci do řízených extension bodů a stabilních rozhraní,[5]
  • a umožnil rychlejší adopci inovací včetně AI bez toho, aby se každá inovace měnila v technický projekt „zachraňme custom code“.

Důležité očekávání: Clean Core nezruší potřebu upgradů a testů; cílem je, aby upgradování bylo častější a levnější, protože core zůstává „čistý“ a extension architektura upgrade „neblokuje“.[6]

Principy Clean Core jako systémová disciplína pro enterprise architekturu

SAP rámuje Clean Core jako průřez pěti dimenzemi: procesy, rozšiřitelnost, data, integrace a provoz. Pro zkušeného architekta je zásadní vnímat, že „Clean Core pro extensibility“ je jen jedna dimenze – bez disciplíny ve zbytku se technický dluh vrátí „oklikou“.[7]

Procesy – preferovat fit‑to‑standard a maximalizovat využití standardních best practices; diferenciaci dělat jen tam, kde má měřitelnou business hodnotu.[8]

Extensibility – rozšíření mají být cloud‑kompatibilní, oddělená od standardního kódu a integrovaná přes stabilní rozhraní; v ideálu „BTP‑first“ (tj. preferovat side‑by‑side, pokud není nutná těsná vazba na core).[9]

Data – „clean core potřebuje clean data“: pokud data governance selhává, často se kompenzuje logikou v custom kódu a integracích. SAP explicitně zmiňuje potřebu spoléhat se na silnou kvalitu dat jako součást motivace Clean Core.[10]

Integrace – cílem je spolehlivá, flexibilní krajina s minimalizací integrační komplexity a customizací; jinak se integrace stává zdrojem „integration debt“.[8]

Operations – provozní model má být efektivní, s governance strukturami, housekeeping procesy, a (prakticky) s automatizovanými quality gates a měřením Clean Core KPI.[11]

Klíčový důraz: pro každou technickou dimenzi Clean Core je potřeba silná governance – nejen doporučení pro vývojáře.[12]

Extensibility architektura napříč edicemi S/4HANA

Základní volby: in‑app, on‑stack developer, side‑by‑side

SAP dlouhodobě rozlišuje tři hlavní způsoby rozšiřování S/4HANA:

  • In‑App Extensibility (Key User) : rozšíření přes Fiori „key user“ nástroje (typicky UI adaptace, custom fields, některé BAdI/logic, custom BO apod.).[13]
  • On‑Stack Developer Extensibility (ABAP Cloud / embedded developer extensibility) : vývoj v ABAP na stacku S/4HANA, ale pod pravidly ABAP Cloud (released APIs, released extension points, upgrade‑stabilita).[14]
  • Side‑by‑Side Extensibility (SAP BTP) : aplikace/servisy běží mimo core (na BTP), integrace probíhá přes stabilní API a/nebo business events; side‑by‑side má být nezávislé na verzi S/4HANA a vyžaduje remote komunikaci.[15]

SAP pro rozhodování doporučuje pokrýt use‑case často kombinací on‑stack i side‑by‑side (hybrid), pokud to dává architektonický smysl.[16]

Co se liší mezi on‑premise, private cloud a public cloud

Rozdíly mezi edicemi nejsou jen licenční; jsou hlavně governance a povolené techniky :

  • V public cloud je Clean Core vynucený: pravidla typu „no modifications“, „pouze stabilní veřejná API a extension points“ jsou povinná.[17]
  • V private cloud a on‑premise je Clean Core doporučený (ne vždy povinný), ale SAP explicitně doporučuje „řídit se pravidly public cloud kdykoli je to možné“, protože upgrade projekty nezmizí a cílem je snížit jejich dopad.[18]
  • ABAP Cloud jako vývojový model je dostupný napříč edicemi – a je povinný v public cloud pro nové zákazníky od 2208 , zatímco v private/on‑premise je doporučen jako „clean core extensibility model“.[19][20]
  • Release cadence: ABAP Cloud inovace přichází kvartálně v BTP ABAP environment, půlročně v public cloud a řádově pomaleji v private/on‑premise (SAP uvádí rozdílné cykly).

Praktický dopad pro landscapy: stejný „business requirement“ může mít v public cloud jediné proveditelné řešení (side‑by‑side / released extension points), zatímco v on‑premise se dá „udělat cokoliv“ – ale za cenu horší budoucí udržitelnosti a rizika Level C/D dluhu.[21]

Technické stavební kameny a extensibility model

ABAP Cloud jako „guardrails“ pro Clean Core

ABAP Cloud je cloud‑ready vývojový model, který omezuje jazyk a přístup k objektům tak, aby custom logika používala pouze released APIs a tím byla upgrade‑stabilní. V praxi to znamená:[22]

  • používáte omezenou jazykovou verzi („ABAP for Cloud Development“) a přístup k repository objektům je omezen na objekty se stavem release (released API).[23]
  • v ABAP Cloud se očekává RAP jako transakční programovací model pro vytváření business apps/services/extensions (Fiori/OData), nikoli klasický „procedurální report + BAPI + update tabulek“.[24][25]
  • vývoj je postaven na ADT a release state („API State“) a release kontrakty (např. C0/C1) jsou explicitní součástí lifecycle.

SAP zároveň zdůrazňuje, že vývoj provedený pomocí ABAP Cloud „dodržuje clean core principy by default“.[26]

CDS Views a RAP jako „rozhraní“ k datům a chování

  • CDS view entity je základní stavební blok datového modelu v ABAP CDS a typicky tvoří „contract“/VDM vrstvu, která je bezpečnější než přímé čtení aplikačních tabulek.[27][28]
  • RAP definuje end‑to‑end architekturu pro transakční aplikace v ABAP Cloud: data model, CRUD, business logiku, transakční chování (locking/numbering), a obvykle OData exposure.

Z pohledu Clean Core je podstatné, že RAP/CDS nejsou jen „moderní syntax“, ale způsob, jak vytvářet upgrade‑stabilní a verzovatelná rozhraní pro rozšíření.

SAP BTP: CAP, Kyma, ABAP environment a integrační vrstva

Side‑by‑side přístup staví na tom, že extension běží mimo core a komunikuje přes stabilní API nebo events. BTP pak poskytuje více runtime a modelů:[29]

  • pro pro‑code vývoj (např. CAP) a low/no‑code (SAP Build) – SAP toto uvádí jako typické technologie pro „nové extensions“ v Tier 1/Level A směru.[30][31]
  • pro event‑driven integraci je běžný pattern s Event Mesh; SAP popisuje konfiguraci komunikace mezi S/4HANA Cloud a Event Mesh v Kyma prostředí jako součást kyma scénářů.

Kde jsou „klasické“ enhancementy v Clean Core

SAP model A–D explicitně uvádí jako nejrizikovější (Level D) techniky: modifikace SAP objektů, write access na SAP tabulky a implicit enhancements . ABAP Cloud FAQ navíc přímo říká, že implicit enhancement spots by se měly vyhýbat podobně jako modifikacím.[32][33]

To lze přeložit do praktických pravidel:

  • User exits / implicit enhancements : typicky spadají do Level D (nebo minimálně mimo „clean“ doporučení), protože jsou blízko standardu, křehké a upgrade‑rizikové.[34]
  • BAdI : pokud je BAdI/release extension point oficiálně released (a používáte jej v rámci ABAP Cloud guardrails), míříte k Level A/B; pokud není released a saháte na interní objekty, sklouzáváte do C/D.[35]

Klasifikační rámec A–D a mapování příkladů

SAP „clean core level concept“ rozšiřuje původní 3‑tier model do čtyř úrovní (A–D) a hodnotí rozšíření podle toho, jak je postavené, jak je oddělené od core a jak upgrade‑stabilní je jeho životní cyklus. Úroveň A je cílový stav, ale model je záměrně pragmatický (uznává realitu komplexních landscape).[9][36]

SAP také zdůrazňuje, že dává společný jazyk pro plánování investic do cleanup/modernizace a doporučuje prioritizovat odstranění Level D jako první.[37]

Důležitý detail z praxe (a typická „past“): při kvalifikaci rozšíření se posuzuje celá extension – i když je 90 % moderní, jeden Level D vzor ji může „shodit“. SAP Learning přímo uvádí princip: kvalifikace podle nejnižší použité technologie v extension.[38]

Tooling a assessment: jak identifikovat „dirty core“

Co je „dirty core“ v terminologii SAP

Prakticky lze „dirty core“ číst jako přítomnost Level D (a často i Level C) stop: modifikace, implicitní enhancements, write access na SAP tabulky, použití objektů označených jako noAPI, nebo přímé závislosti na interních SAP objektech.[32]

Základní nástrojová sada pro měření a plánování remediation

ABAP Test Cockpit (ATC) + Cloudification Repository Cloudification Repository poskytuje seznam released i non‑released objektů včetně successorů a ATC jej využívá v checku „Usage of Released APIs (Cloudification Repository)“ pro analýzu custom code napříč ECC/S/4HANA.[42]

Centralní governance přes ATC na BTP SAP (v materiálech k Clean Core extensibility) doporučuje ATC na BTP jako centrální nástroj pro governance (zejména pro private cloud a on‑premise) a zmiňuje konkrétní provozní schopnosti: custom code analýzy, práce s exemptions, baseline pro legacy code, plánování ATC běhů, a vazbu na transport/ change management.[43]

Custom Code Migration app Oficiální guide popisuje, že aplikace umí scope & analyzovat custom code pro migraci na S/4HANA, umí identifikovat nepoužívaný kód na základě usage dat a vytvořit mazací transport pro odstranění nepoužívaného ABAP source code během konverze. Stejný guide také popisuje variantu, kdy jako „central check system“ funguje BTP ABAP environment a analyzuje on‑premise kód vzdáleně přes Cloud Connector; BTP ABAP environment má kvartální aktualizace a obsahuje nejnovější funkce CCM.[44][45]

SAP Readiness Check SAP Readiness Check umí vizualizovat dopady custom code a lze do něj nahrát výsledky ATC analýzy (ATC findings pak nahradí „high level“ výsledky v dashboardu). SAP navíc popisuje „Enhanced custom code analysis“ pro lepší viditelnost ATC findings vůči cílové S/4 verzi.[46][47]

Doporučený „assessment workflow“ pro enterprise architekta

  • Inventarizace a klasifikace : získejte seznam Z* a add‑on objektů + určete jejich A/B/C/D profil pomocí ATC + Cloudification Repository (a/nebo CCM projekty).[48]

2.[43]

  • Baseline a exemptions politika : pro legacy code nastavte baseline tak, aby governance netrestala „historii“, ale vytvářela tlak na nové změny; zároveň stanovte politiku exemptions (kdo schvaluje, expirace, evidence). Prioritizace : jako první řešte Level D (nebo D‑stopy v kritických objektech), poté nejrizikovější C. To odpovídá doporučení SAP zaměřit se nejprve na highest‑risk extensions.[32][49]
  • Roadmapa „B → A“ : pro B‑profil (classic APIs) si zaveďte backlog na postupné nahrazování successor/released API, jakmile jsou dostupné.

Dopad na vývoj, governance a DevOps model

Změna mindsetu: z ECC „dělej to kdekoliv“ na S/4 „dělej to v povolených vrstvách“

V ABAP Cloud se vývoj posouvá k explicitnímu contract‑driven přístupu:

  • nejdřív hledám released API / released extension point,
  • teprve pak píšu kód (RAP/CDS),
  • a dotýkám se standardu jen přes stabilní kontrakty.

SAP také zdůrazňuje, že v public cloud je nutné držet se pravidel: žádné modifikace SAP objektů a používání jen stabilních veřejných API/extension points.[51]

Doporučené vzory a anti‑vzory

Doporučené vzory

  • RAP BO nad custom tabulkami + exposure přes OData (ideálně v ABAP Cloud guardrails) jako standardní transakční přístup.[52]
  • Wrapper/facade pattern : pokud chybí released SAP API, izolujte volání classic API do kontrolované vrstvy (Cloud API enablement / wrapper objekty), aby zbytek aplikace zůstal v Level A boundary. SAP sample explicitně ukazuje mitigaci chybějícího released API tím, že se obalí vhodné classic API (např. BAPI_PR_CREATE) a použije se v Level A aplikaci.[41]
  • Side-by-side „data product / process extension“ pro scénáře s vlastním lifecyklem, vlastní škálou, nebo integrací více systémů – side‑by‑side má být nezávislé na verzi S/4HANA a běží mimo core.[53]

Zakázané / důrazně nedoporučené

  • write access do SAP aplikačních tabulek (přímé UPDATE/INSERT do standardních tabulek) – SAP to uvádí jako Level D.[32]
  • implicit enhancements a modifikace standardu – rovněž Level D.[55]
  • neřízené volání interních objektů (třídy/tabulky/FMs mimo classic/released seznam) – typicky Level C s upgrade rizikem.

CI/CD a transportní strategie jako součást Clean Core

Clean Core governance není jen „code style“. Pokud nemáte CI a transportní disciplínu, vznikne „operational debt“ a do core se dostane Level C/D chování i při dobrých úmyslech.[56]

Transport a změnové řízení

  • Public cloud pracuje s 3‑system landscape a řízením transportů pro customizing, key user a developer změny; existuje k tomu oficiální SAP dokumentace.[57][58]
  • Pro moderní ABAP lifecycle existuje gCTS (Git‑enabled Change and Transport System), který propojuje CTS procesy s Git jako externím version managementem.[59]
  • V prostředí BTP se používá SAP Cloud Transport Management pro transport deliverables mezi BTP účty/runtimy (včetně ABAP prostředí).[60]
  • abapGit je oficiálně popsaný jako nástroj pro přenos source code assets mezi on‑premise a BTP ABAP environment (side‑by‑side vývoj).

DevOps praktika pro ABAP environment na BTP SAP Learning uvádí, že pro ABAP na BTP je potřeba kombinovat Git flexibilitu s ABAP CTS, doporučuje Cloud Transport Management pro automatizovaný transport a zároveň upozorňuje, že pro CI pipeline se typicky používá third‑party (např. Jenkins).[61]

Trade‑offs, migrační strategie, scénáře a budoucí roadmap

Kdy je Clean Core těžké nebo „nepohodlné“

Chybějící released APIs / gaps ve standardu Nejčastější reálný problém: business potřebuje rozšířit proces, ale neexistuje vhodné released API/ extension point. SAP proto pracuje s pragmatickou cestou: Level B (classic APIs) a wrapper strategie jako přechodový most do Level A, až bude successor dostupný.[62]

Výkon a latence u side‑by‑side Side‑by‑side je upgrade‑výhodné, ale vyžaduje remote komunikaci a integraci přes API/events, což přináší latenci, monitoring a integrační náklady; SAP Learning explicitně říká, že side‑by‑side je mimo core a remote komunikace je nutná.[63]

Cena a organizační dopad BTP BTP‑first strategie znamená, že část diferenciace přesunete na platformu s vlastním lifecycle, runtime a governance; v praxi přibude cost center, skills (CAP/Kyma/integration), provozní dohled. SAP nicméně současně prezentuje BTP jako základ pro side‑by‑side a ABAP Cloud jako on‑stack model, který drží clean core pravidla.[64]

Migrační strategie z „dirty core“ na Clean Core

Níže je praktický postup, který odpovídá logice SAP rámců: Clean Core je průběžná strategická aktivita a součást metodik (success plan, quality gates), nikoli jednorázový refactoring sprint.[65]

Krokový postup 1. Nastavte cílový stav a policy : definujte, co je pro vás Level A/B/C/D „tolerovaný“ stav, včetně doby expirace pro C/D výjimky. (SAP rámec umožňuje pragmatickou cestu, ale Level A je ideál a Level D je nejvyšší riziko.) 2. Zaveďte měření : ATC + Cloudification Repository jako povinný quality gate; pro landscape s více systémy centralizujte governance (ATC na BTP). 3. Racionalizujte custom code : použijte CCM scoping s usage daty a odstraňte nevyužívané objekty; minimalizujete objem, který musíte modernizovat. 4. Eliminujte Level D : modifikace/implicit enhancements/write access na SAP tabulky řešte jako blockers (redesign na released extension points, eventing, side‑by‑side, nebo wrapper). 5. Stabilizujte Level C : kde je C objektivně nutné (např. chybí API), udělejte z toho řízený „debt register“: dokumentace, test scope, upgrade playbook, owner. (SAP model C výslovně zmiňuje upgrade riziko a potřebu včasné identifikace změn.) 6. Budujte nový vývoj Level A : nové požadavky už neimplementujte „legacy style“; vynucujte ABAP Cloud/RAP a side‑by‑side, aby se dluh dál nezvětšoval.[32][66][67][68][69]

Vztah k brownfield/greenfield/selective data transition RISE metodika výslovně pracuje s různými „transition journeys“ (new implementation, system conversion, selective data transition). Clean Core policy (quality gates, success plan) by měla být konzistentní napříč těmito přístupy; liší se jen to, kolik legacy dluhu přenášíte.[70][71]

Praktické scénáře s jasným ANO/NE

Definice: „Clean‑core compliant“ zde beru jako SAP kategorizaci, kde A a B jsou compliant , C je částečně compliant a D není doporučeno.[72]

Pokud je implementace v ABAP Cloud guardrails a integrace do standardu jde přes released API/extension points, jde o typický Level A on‑stack extension.[73]

Clean Core klasifikace primárně řeší vztah k SAP core. Zápis do vlastních tabulek není sám o sobě „dirty core“. Riziko je spíš doménové: obcházíte validace/locking/determinations a vytváříte nekonzistentní „backdoor“. Doporučení: držet jedinou transakční cestu přes RAP BO.[74]

Freestyle Fiori aplikace konzumující vaše custom OData

UI technologie neklasifikuje clean core; rozhoduje, jaký je kontrakt OData a co je pod ním. Pokud je OData postavené nad released rozhraními (ideálně RAP/OData V4, případně kontrolovaně classic API), zůstáváte v A/B.[39]

Budoucí outlook: proč je Clean Core kritický pro roadmap a AI

SAP přímo propojuje Clean Core s rychlejší adopcí inovací a „easier adoption of AI“. Dává to technický smysl:[75]

  • AI asistenti a agenti jsou nejefektivnější tam, kde mají konzistentní procesy, stabilní API kontrakty a čistá data – což je přesně to, co Clean Core prosazuje.[76][77]
  • SAP zároveň uvádí, že Joule (AI copilot) podporuje produktivitu vývojářů (generování „clean“ kontextového kódu, automatizace) jako součást ekosystému okolo clean core strategie.
  • V ABAP kontextu SAP zmiňuje plánované AI use‑cases pro custom code transformation (AI‑based vysvětlení legacy kódu, vysvětlení ATC findings a návrhy adaptace) integrované do ADT – což je[78]
  • prakticky přímá vazba „Clean Core governance ↔ AI akcelerace remediation“.

Zjednodušeně: čím více se S/4HANA inovace (včetně AI) posouvají do „continuous“ režimu, tím méně je prostor pro Level C/D dluh – a tím více Clean Core přechází z „best practice“ na podmínku udržitelného provozu .[79]
