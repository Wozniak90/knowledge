1. Kontext — proč vůbec řešit

LLM modely mají dvě zásadní omezení:

  • Knowledge cutoff — model byl trénován do určitého data, novější znalost nezná
  • Kontextové okno — model vidí jen to, co mu předáme; nemá „paměť" mimo kontext

Aby LLM dokázal zodpovídat otázky na základě firemních/privátních nebo aktuálních dat, existují tři hlavní přístupy:

PřístupPrincipKdy vhodné
Fine-tuning Dotrénovávání modelu na vlastních datech Styl, formát, stabilní znalost domény
Long-context window Vložit celý dokument do kontextu Malý, statický dataset
RAG Dynamicky vyhledat relevantní kusy a injektnout do promptu Velká, měnící se KB; enterprise data

2. RAG architektura

Ingestion pipeline (offline — příprava dat)

graph LR D[Dokumenty / PDF / web] --> CH[Chunking] CH --> EM[Embedding model] EM --> VS[(Vector Store)]

Klíčové kroky:

  1. Chunking — rozřezání dokumentů na menší části (chunk size = kompromis: moc velké = irrelevantní kontext, moc malé = ztratí kontext)
  2. Embedding — každý chunk převeden na numerický vektor (sémantická podobnost)
  3. Vector Store — databáze vektorů (Pinecone, Weaviate, Chroma, SAP HANA Cloud)

Retrieval pipeline (online — za běhu)

graph LR Q[User Query] --> QE[Embed query] QE --> SS[Similarity search] SS --> VS[(Vector Store)] VS --> RC[Relevantní chunky] RC --> AUG[Augmented prompt] AUG --> LLM[LLM] LLM --> ANS[Odpověď + citace]

Klíčové kroky:

  1. Embed query — dotaz uživatele převeden na vektor
  2. Similarity search — hledání nejbližších vektorů (cosine similarity / L2 distance)
  3. Augmentation — relevantní chunky vloženy do systémového promptu
  4. LLM — generuje odpověď grounded v reálných datech

3. Je RAG ještě potřeba?

Argument PROTI RAG: Long-context LLMs

  • Modely jako Gemini 1.5 Pro mají kontext 1M+ tokenů — lze vložit celé knihy
  • Proč se vůbec obtěžovat s retrieval?

Proč RAG stále vítězí

AspektLong-contextRAG
Náklady🔴 Velmi drahé (každý token)🟢 Platíš jen za relevantní část
Latence🔴 Pomalejší s velkým kontextem🟢 Rychlejší inference
Aktuálnost🟡 Musíš přegenerovat každý call🟢 KB se updatuje nezávisle
Škálovatelnost🔴 KB musí pasovat do okna🟢 Neomezená velikost KB
Přesnost🟡 Lost-in-the-middle problém🟢 Fokusovaný kontext
Citace/zdroje🔴 Náročné🟢 Přirozené (znáš zdroj chunku)
Závěr videa: RAG není zastaralý — je stále nejpraktičtější přístup pro enterprise use cases s velkou, dynamickou knowledge base.

4. Fine-tuning vs RAG

Fine-tuningRAG
Co trénujeVáhy modeluNic — runtime retrieval
Vhodné proStyl, tone, formát, doménový slovníkFaktické znalosti, aktuální data
Náklady🔴 Vysoké (GPU, čas)🟡 Střední (embedding + vector DB)
Aktualizace dat🔴 Re-train🟢 Stačí aktualizovat KB
Halucinations🔴 Neřeší🟢 Grounding snižuje

Doporučení: V praxi se kombinují — fine-tuned model + RAG = nejlepší výsledky.

5. Výzvy a best practices RAG

Chunk size

  • Příliš velký chunk → obsahuje irelevantní info → LLM "zmatený"
  • Příliš malý chunk → ztratí kontext → nízká kvalita odpovědi
  • Best practice: experimentovat, typicky 256–512 tokenů + overlap (~20%)

Relevance scoring

  • Top-K retrieval: vrátit K nejpodobnějších chunků (typicky 3–5)
  • Hybrid search: kombinace vector search + keyword search (BM25) pro lepší přesnost

Halucinations

  • RAG snižuje ale nevylučuje halucinations
  • Mitigation: instrukce „odpovídej jen na základě poskytnutého kontextu, jinak řekni nevím"

Evaluace RAG systému

  • Context relevance — jsou retrievnuté chunky k dotazu relevantní?
  • Faithfulness — je odpověď věrná chunktům?
  • Answer relevance — odpovídá odpověď dotazu?
