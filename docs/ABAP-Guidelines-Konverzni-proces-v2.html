1. Proč vůbec měnit guidelines

Tři vnější tlaky (Ivan Radulovic, SAP EMEA/MEE webinář)

Zákazníci ke změně netlačíme z ideologických důvodů — tlačí je tři reálné síly:

1. Rychlost byznysových změn

Supply chain disruption, AI adopce, nové regulace — silně customizovaný ERP neumí reagovat rychle. Každá změna procesu = projekt. Každý projekt = riziko.

2. Uzavřená cesta k AI (Joule)

SAP Joule je trénovaný na SAP standardu. Systém s tisíci modifikacemi a Z-objekty, které přepisují core logiku, nebude Joule umět používat. Špatná data nebo modifikovaná logika = AI halucinace. Clean Core je předpoklad pro SAP AI.

3. Narůstající komplexita — cloud lock-in

S/4HANA Public Cloud (SaaS) odmítá zákazníky s D-level kódem. Pokud chce zákazník přejít na SaaS, musí nejdříve svůj kód přiblížit standardu. Každá modifikace je technický dluh, který bude dražší čím déle čekáme.

Naše přidaná hodnota jako partner

Legacy guidelines nebyly špatné pro svůj čas — byly optimální pro ECC/GUI svět. Svět se změnil. Naší rolí není kritizovat minulost, ale pomoct zákazníkovi pochopit:

  • Kde jsou jeho guidelines zastaralé
  • Co konkrétně má změnit a v jakém pořadí
  • Jak na to nástroji, které SAP dnes nabízí

3. Rozhodovací rámec — čím se řídíme

Tři otázky pro každé staré pravidlo

Při každém legacy pravidle jsme si kladli:

  1. Funguje toto v S/4HANA Public Cloud?

Pokud ne → typicky vyžaduje změnu při cíli Public Cloud. Pro čistě on-premise zákazníka bez cloud plánu: vědomý technický dluh s upgrade-risk flaggingem.

  1. Je závislé na SAP interním objektu bez stability kontraktu?

Pokud ano → Level C nebo D → doporučeno mít plán migrace.

  1. Blokuje toto adopci AI nástrojů (Joule, SAP Build)?

Pokud ano → prioritizovat jako high-impact pro zákazníka s AI roadmapou.

Pragmatický filtr

Nepřepsali jsme vše najednou. Rozdělili jsme na:

4. Pravidlo po pravidle — co a proč se změnilo

4.1 Přístup k datům

🔴 POVINNÉ — Public Cloud | 🟡 DOPORUČENO pro nový vývoj on-premise

🟢 TOLEROVÁNO pro existující SELECT FROM BKPF/BSEG v on-premise bez cloud plánu — přechod při dotčení objektu.

Staré pravidlo: SELECT FROM BKPF JOIN BSEG ON...

Nové pravidlo: SELECT FROM I_JournalEntry

Proč:

BKPF a BSEG jsou SAP interní tabulky — bez stability kontraktu. SAP je v S/4HANA restrukturalizoval (Universal Journal — ACDOCA). Zákaznický kód psaný na BKPF/BSEG buď selže nebo vrátí nesprávná data na S/4HANA. Released CDS view I_JournalEntry abstrahuje tuto změnu — kód zůstane funkční i po datové restrukturalizaci.

Jak jsme to ověřili:

ATC S/4 Readiness Check na pilotním projektu ukázal SELECT FROM BKPF jako Error (Level D). Custom Code Migration App nabídla quick fix na I_JournalEntryItem.

4.2 Uživatelské rozhraní

🔴 POVINNÉ — Public Cloud (WRITE: statement nefunguje, SAP GUI není podporováno)

🟢 TOLEROVÁNO pro existující ALV/Dynpro reporty u zákazníka bez cloud plánu — žádný nový vývoj.

Staré pravidlo: Dynpro screens + WRITE: statement + ALV Grid

Nové pravidlo: CDS + OData V4 + Fiori Elements

Proč:

SAP GUI neznamená konec světa v on-premise. Ale:

  • Public Cloud SAP GUI nepodporuje vůbec
  • WRITE: statement je deprecated v ABAP for Cloud language version
  • Fiori je jediné UI podporované ve všech deployment modelech (on-prem, private cloud, public cloud)
  • Fiori Elements s CDS anotacemi = UI generované automaticky — méně práce, víc standardu

Pragmatická poznámka:

Existující WRITE: reporty v on-premise, které zákazník denně používá a neplánuje migraci do cloudu — nemigrovat hned. Investice se vrátí až při cloud přechodu. Ale žádný nový vývoj nesmí WRITE: používat.

4.3 Rozšiřitelnost

🔴 POVINNÉ — Public Cloud pro nový vývoj (implicit enhancement zakázán)

🟡 DOPORUČENO pro nový vývoj on-premise

🟢 TOLEROVÁNO pro existující User-Exits kde released BAdI náhradník neexistuje — Level C s changelog monitoringem.

Staré pravidlo: User-Exit jako první volba, implicit enhancements pro „nenápadné" změny

Nové pravidlo: BAdI jako první volba, implicit enhancements zakázány

Proč:

User-Exits jsou technicky funk ční ale:

  • SAP je aktivně odstraňuje bez náhrady (SAP Note 3414378)
  • Implicit enhancements jsou po upgradu neviditelné — vývojář neví, že tam jsou, dokud se něco nerozbije
  • BAdI je formalizovaný kontrakt — SAP garantuje zpětnou kompatibilitu released BAdI

Kde jsme narazili na limit:

Pro část funkcionalit released BAdI náhradník prostě neexistuje. V takovém případě:

  1. Zkontroluj SAP Influence program — požádej o release
  2. Použij Level C s changelog monitoringem, zdokumentuj jako technical debt
  3. Hlídej ATC výsledky před každým upgradem

4.4 Integrace

🟡 DOPORUČENO pro nový vývoj

🟢 TOLEROVÁNO pro existující RFC integrace kde protistrana REST nepodporuje — BAPI jako Level B přechodné řešení.

Staré pravidlo: RFC jako primární mechanismus integrace, IDoc pro async

Nové pravidlo: HTTP/OData REST jako primární, SAP Integration Suite pro orchestraci

Proč:

RFC je proprietární SAP protokol — funguje jen SAP-to-SAP. Každý moderní systém mluví HTTP. Released OData API na api.sap.com jsou:

  • Dokumentovány
  • Versionovány (V2, V4)
  • Dostupné z jakéhokoliv tech stacku (BTP, Node.js, Python, mobilní app)

Zachováváme BAPI:

Kde existující BAPI nebyly nahrazeny released REST API, zůstávají jako Level B přechodné řešení. Nevolat přes RFC destination pokud možno — obalit do CAP / Integration Suite proxy.

4.5 Function Modules → ABAP Classes

🔴 POVINNÉ — Public Cloud (FORM rutiny a nové FM zakázány v ABAP Cloud language version)

🟡 DOPORUČENO pro nový vývoj on-premise

🟢 TOLEROVÁNO pro existující Z-FM volané pouze lokálně — migrovat postupně.

Staré pravidlo: Function Module jako zapouzdření logiky

Nové pravidlo: ABAP Class s Interface jako zapouzdření logiky

Proč:

Function Modules v ABAP Cloud Development language version nelze použít pro on-stack clean core vývoj (jen global class FM přes released API). ABAP třídy podporují:

  • Dependency Injection — testovatelnost
  • Polymorfismus přes Interface — vzájemná zaměnitelnost
  • Finalization / Singleton vzory

Zachováváme existující Z-FM:

Existující Z-FM které jsou volány pouze lokálně a pracují se standardním ABAP — Level B nebo C. Migrovat postupně při každém dotčení objektu.

4.6 Testování

🟡 DOPORUČENO — silně doporučeno pro každý nový nebo refaktorovaný kód; bez testů je bezpečný refaktoring nemožný.

Staré pravidlo: Unit testy doporučené, v praxi vzácné

Nové pravidlo: Unit testy povinné pro každou třídu s business logikou

Proč:

Refaktoring legacy kódu na Clean Core je bezpečný jen s testy. Bez testů nevíme, zda přepsaný kód dělá totéž co původní. ATC quality gate s unit test variantou blokuje transport objektu bez testů.

4.7 Autorizace a bezpečnost

🟡 DOPORUČENO pro nový vývoj s CDS/RAP vrstvou

🟢 TOLEROVÁNO existující AUTHORITY-CHECK v legacy kódu — funkční, bezpečný, Level B.

Staré pravidlo: AUTHORITY-CHECK OBJECT '...' ID 'ACTVT' FIELD '...' rozptýlené v kódu; přístup k datům filtrovaný ABAP podmínkami

Nové pravidlo: CDS Access Control (DCL) pro datovou vrstvu; AUTHORITY-CHECK jen pro released autorizační objekty ve vystavených RAP akcích

Proč:

V Clean Core modelu je autorizace záležitostí datové vrstvy — CDS anotace @AccessControl.authorizationCheck: #CHECK s DCL podmínkami. Výhody:

  • Autorizace deklarativní a centralizovaná — ne rozptýlená v každém selectu
  • DCL se aplikuje automaticky na všechny konzumenty CDS view (Fiori, OData, direct ABAP)
  • Konec scénáře: „zapomněl jsem AUTHORITY-CHECK v tomto selectu"

Kde jsme narazili na limit:

Ne každý autorizační objekt je Released API. Zákaznické autorizační objekty (SU21) jsou samozřejmě v rukou zákazníka a AUTHORITY-CHECK na ně je legitimní i v Clean Core.

4.8 Tiskové formuláře a output management

🟡 DOPORUČENO pro nový vývoj

🟢 TOLEROVÁNO existující SAPScript/SmartForms v on-premise dokud zákazník neplánuje cloud — migration je náročná na specialistu, viz sekce 7.

Staré pravidlo: SAPScript (transakce SE71) nebo SmartForms (SF01) pro tisk a PDF výstupy

Nové pravidlo: Adobe Document Services (ADS) přes SAP BTP; nové formuláře ve SFP (Adobe Forms) nebo Output Management Framework

Proč:

SmartForms jsou v SAP Cloud ERP v maintenance mode — SAP nepřidává nové funkce, neinterprétuje je v Public Cloud verzi. Adobe Forms (SFP) a cloudový ADS jsou aktivně rozvíjené. Navíc:

  • ADS produkuje PDF/A a digitálně podepsatelné dokumenty — regulatorní požadavky
  • Output Management Framework (BTP) odděluje template od triggeru — lepší governance

Reálný kontext:

Patrick Weber ve svém 8měsíčním projektu identifikoval SAPScript/SmartForms migraci jako jednu z nejnáročnějších — tým neměl dostatek formulářové expertízy (ve svém blogu SAP Community 2024, díl 2). Doporučujeme tuto oblast plánovat s delším časovým horizontem a případně zapojit specialistu na ADS.

4.9 Workflow a business procesy

🟡 DOPORUČENO pro nový vývoj

🟢 TOLEROVÁNO existující SAP Business Workflow — Level C, ne D; nemigrovat násilně.

Staré pravidlo: SAP Business Workflow (SWDD, SWIA) nebo workflow v pozadí přes vlastní stavové automaty

Nové pravidlo: SAP Build Process Automation (BTP) pro nové procesy; SAP Flexible Workflow pro jednodušší S/4 integrované scénáře

Proč:

SAP Business Workflow je v maintenance mode — SAP přestalo přidávat nové funkce a aktivně zákazníky nasměrovává na BTP Process Automation. Klíčový argument:

  • Workflow jako cloud service = žádná závislost na ABAP stack upgradu
  • Low-code tvorba procesů (i non-ABAP administrátor může upravit workflow)
  • Integrace s Teams, e-mailem, SAP Build Work Zone nativně

Pragmatická poznámka:

Existující SAP Business Workflow zákazníkovi nemigrujeme násilně — je to Level C, ne Level D. Ale nové schvalovací procesy a procesy spuštěné po Clean Core projektu navrhujeme vždy na BTP PA.

4.10 Background jobs a batch processing

🔴 POVINNÉ — Public Cloud (JOB_OPEN/CLOSE zakázány v ABAP Cloud language version)

🟡 DOPORUČENO pro nový vývoj on-premise.

Staré pravidlo: SM36/SM37 + klasický ABAP program spuštěný jako job; JOB_OPEN / JOB_SUBMIT / JOB_CLOSE volání

Nové pravidlo: SAP Application Job Framework — interface IF_APJ_RT_RUN, definice jobů přes CL_APJ_DT_CREATE_CONTENT; scheduling přes Application Jobs UI (Fiori)

Proč:

Klasické SM36 joby mají technická omezení v cloud deploymentu:

  • JOB_OPEN/CLOSE funkce jsou v ABAP for Cloud language version zakázány (Level D)
  • Application Job Framework je released API — IF_APJ_RT_RUN a IF_APJ_DT_EXEC_OBJECT
  • Fiori-based scheduling UI = zákazník nepotřebuje BASIS přístup pro správu jobů
  • V BTP ABAP Environment je Job Scheduling Service dostupný jako managed service

Kde jsme narazili na limit:

Zákaznické Z-programy volané jako joby typicky obsahují desítky SELECT FROM unreleased tabulek — migrace jobu = migrace celého programu. Plánovat jako součást datové vrstvy migrace (sekce 4.1).

4.11 Exception handling

🟡 DOPORUČENO — sy-subrc je funkční, ale náchylný k přepsání v komplexním kódu; class-based exceptions povinné v ABAP Cloud.

Staré pravidlo: sy-subrc kontrola po každém volání; EXCEPTIONS parametr na FM voláních; MESSAGE ... TYPE 'E' pro chyby

Nové pravidlo: Class-based exceptions cx_* s RAISE EXCEPTION TYPE; TRY...CATCH...ENDTRY bloky; vlastní zákaznické exception třídy dědící z cx_static_check nebo cx_dynamic_check

Proč:

sy-subrc je globální proměnná — na prvním volání ji může přepsat jiný kód. Je to architektonická past. Class-based exceptions:

  • Přenáší context (message, cause, previous) — debuggování rychlejší
  • ABAP for Cloud language version vynucuje class-based exceptions pro released API volání
  • code-pal-for-abap check PREFER_RAISE_EXCEPTION_NEW flaguje starý styl automaticky v ATC

Reálný příklad:

Cloudification Repository abap-atc-cr-cv-s4hc označuje FM_FUNCTION_MODULE_NAME (vrací sy-subrc) jako BF9 — avoid. Nástupce je cl_abap_module_names class-based API.

4.12 Kódová kvalita a styl

🔴 POVINNÉ — Public Cloud (FORM rutiny zakázány v ABAP Cloud)

🟡 DOPORUČENO code-pal-for-abap pro nový kód — nástroj pro zlepšení, ne soudce.

Staré pravidlo: Žádný vynucený styl; FORM routiny (FORM ... ENDFORM) jako organizační jednotka kódu; komentáře v češtině nebo němčině

Nové pravidlo: Clean ABAP styleguide (GitHub: SAP/styleguides) vynucený přes code-pal-for-abap (62+ ATC checks); výhradně ABAP OO třídy; angličtina pro identifikátory

Proč:

Clean ABAP styleguide SAP vydal jako otevřený standard (2019, průběžně aktualizován). FORM rutiny jsou v ABAP for Cloud language version zakázány — nelze vytvořit nové. code-pal-for-abap automatizuje styl jako ATC check:

  • 62+ pravidel pokrývá naming, struktura, OO principy, exception handling
  • Cloud Edition (code-pal-for-abap-cloud) pro ABAP Environment na BTP
  • Default severity: Notification (Priority 3) — nablokuje transport ale reportuje stav

Reálný kontext:

Patrick Weber ve svém projektu nastavil code-pal jako povinný ATC a systém okamžitě hlásil 2,4 milionu finding na 30 let legacy kódu. Klíčové poučení: nespouštět na celém systému najednou — vždy omezit scope na nový nebo migrovaný kód.

5. Co jsme nechali beze změny (a proč)

Naming konvence Z/Y prefix

Zachováno. S/4HANA i ABAP Cloud stále používají customer namespace s Z/Y. Prefix se doplnil o strukturované přípony (ZI_, ZC_, ZCL_) ale základní Z namespace zůstává.

Classic ALV pro existující Z-reporty (on-premise)

Zachováno pro existující kód. Zákazníci mají stovky ALV reportů. Přepsat je všechny najednou na Fiori je nerealistické. Mají-li zákazníci plán zůstat on-premise, nemigrujeme funkční ALV.

BAPI pro integrace bez released REST alternativy

Zachováno jako Level B. Část BAPIs nemá released OData náhradu. Dokud SAP nereleasuje, BAPI zůstávají legitimní — ale s monitoringem Cloudification Repository.

DDIC slovník — domény, datové elementy, struktury zákaznických tabulek

Zachováno. ABAP Dictionary objekty (domény, datové elementy, struktury, zákaznické Z-tabulky) jsou Clean Core beze změny — zákaznická tabulka v Z namespace patří zákazníkovi. Přesouváme jen to, jak k datům přistupujeme (Section 4.1), ne jak jsou data definována v DDIC.

Pozor na: Výjimkou jsou DDIC objekty odkazující SAP interní struktury (INCLUDE structure do BKPF atp.) — ty jsou Level D.

Interní tabulky a Open SQL základní syntax

Zachováno. LOOP AT ... INTO, READ TABLE ... WITH KEY, APPEND, COLLECT, SORT, DELETE — syntaxe a sémantika beze změny. Open SQL SELECT se rozšířil o nové klauzule (GROUP BY inline, UNION, subquery), ale existující syntaxe je zpětně kompatibilní. Kód čtoucí z released CDS views funguje na všech deployment modelech.

Message classes (SE91)

Zachováno pro existující kód. Zprávy (SE91, MESSAGE ... TYPE) jsou stále platné v on-premise S/4HANA. V ABAP Cloud se preferují class-based exceptions nesoucí message (cx_... s IF_T100_MESSAGE), ale existující message classes nevyžadují okamžitou migraci. KernSeife scoring pro Message Class (MSG) = score 1 — Level B, bez blokovacího efektu.

Customizing / IMG (SPRO) konfigurační vrstvy

Zachováno. Clean Core se týká ABAP kódu a jeho závislostí — ne konfiguračních nastavení zákazníka. Customizing přes SPRO/IMG, parametrizace přes tabulkové hodnoty a konfigurační objekty jsou beze změny. Zákaznické konfigurace jsou naopak explicitně podporovány jako „Customizing Extensibility".

Transport management a package struktura

Zachováno. Package concept, transport request workflow a CTS jsou beze změny. ABAP Cloud přidává package property ABAP for Cloud Development jako novou dimenzi — ale to je rozšíření, ne nahrazení. Zákazník nepotřebuje přeorganizovat transport workflow, jen přiřadit správné package atributy novým objektům.

5a. Kde Clean Core přístup nedává smysl

Tato sekce záměrně hraje advocata diaboli. Clean Core je správný směr — ale není odpověď na všechny problémy.

Ekonomicky nevýhodný přepis

Přepis legacy kódu typicky nedává smysl, pokud:

  • Objekt je volán < 10× ročně (UPL data) — technický dluh je teoretický, riziko nízké
  • Zákazník nemá cloudový plán na 5+ let — investice do přepisu se nevrátí
  • Náklad přepisu > 3× náklad jednoho upgradu s tímto objektem — ekonomicky neobhajitelné
  • Kód řeší business kritickou logiku bez unit testů — přepis bez testů je riziko, ne zlepšení

Kde SAP zatím nenahradil legacy API

SAP Released API pokrytí není uniformní. Oblasti kde je Level C vědomé a legitimní rozhodnutí:

  • HR/HCM infotypy (PA0001–PA0xxx) — released náhrada chybí pro velkou část scénářů; SuccessFactors vyžaduje licenci; Level C je vědomé, akceptované rozhodnutí
  • CO interní tabulky (COSP, COSS, COEP) — released CDS views existují ale nepokrývají všechny analytické scénáře
  • Niche FM v legacy modulech (IS-Retail, IS-Banking) — released náhrada neexistuje
  • Část SD pricing (podmínka EXIT_SAPMV45A_*) — Level C, released BAdI s ekvivalentní funkcionalitou neexistuje pro všechny scénáře

Metriky mají slepá místa

KernSeife skóre záměrně neměří:

  • Funkčnost — kód s KernSeife score 0 může být plný bugů
  • Business hodnotu — přepis kódu který nikdo nevolá je ztráta času
  • Upgrade historii — objekt existující 20 let bez SPAU problémů může být stabilnější než nový „clean" kód
  • TCO přepisu — snížení skóre z 100 na 0 může stát miliony, zatímco skóre 15 s nízkým UPL = nulový risk

⚠️ Varování před gamifikací: Nasměrovat tým na „snižte KernSeife skóre" bez byznysového kontextu vede k přepisování kódu který nikdo nevolá. Metriky jsou navigace, ne cíl.

Legitimní on-premise scénář

On-premise zákazník s S/4HANA bez cloudového plánu:

  • ATC findings jsou informace, ne blocker (pokud zákazník nenastaví quality gate)
  • Level D kód je technický dluh — vědomý, zdokumentovaný, s upgrade-risk flaggingem
  • Přechod probíhá při dotčení objektu („boy scout rule") — ne jako standalone projekt
  • KernSeife trend je signál, ne SLA

Klíčové: i pro on-premise zákazníka má Clean Core hodnotu — ale argumentovat Public Cloud nutností je kontraproduktivní. Argumentujte snazší upgrade, lepší testovatelnost, AI readiness — tyto benefity platí i on-premise.

6. Jak implementovat přechod u zákazníka

Fáze 1 — Inventura (2–4 týdny)

Nástroje: SCMON + UPL + ATC S/4 Readiness Check
Výstup:
  - Mapa objektů s Level A/B/C/D klasifikací
  - UPL: co se reálně volá (zombie kód odfiltrovat)
  - Top 10 D-level objektů podle frekvence volání

Fáze 2 — Guidelines a governance (1–2 týdny)

Akce:
  - Prezentovat Legacy × New guidelines (tento dokument + srovnání)
  - Nastavit ATC variant pro Clean Core check
  - Integrovat ATC do CI/CD pipeline nebo transport management
  - Vyškolit development tým (Dev4S4C kurz nebo workshop)

Fáze 3 — Get Clean (průběžně, 6–18 měsíců)

Princip: každý objekt, který se dotýkáš, oprav na Clean Core
  - Nový vývoj: výhradně Level A
  - Bugfix starého kódu: oprav + refaktor na Level A
  - Velký refaktoring: prioritizovat dle UPL frekvence a D-level severity
Metriky: Project KernSeife (ATC automated) + Clean Core Index (Patrick Weber) — měřit týdně

Fáze 4 — Stay Clean (kontinuálně)

ATC quality gate jako permanentní součást transport procesu
Clean Core Index prezentovat managementu čtvrtletně
Sledovat SAP Changelog pro Level C objekty před každým upgradem

Komunikace vůči zákazníkovi

7. Rizika a upřímná varování

⚠️ Released BAdI neexistuje

Část User-Exitů nemá released BAdI náhradu. SAP Note 3414378 dokumentuje exity které SAP odstraňuje, ale ne vždy s náhradou. Doporučení:

  • Level C s changelog monitoringem jako dočasné řešení
  • SAP Influence Program — žádat o BAdI release
  • Dokumentovat jako tech debt s clear upgrade risk

⚠️ ATC quick fix není vždy správný

Custom Code Migration App nabízí quick fixy (přepis SELECT FROM MARA na I_PRODUCT). Ale:

  • Quick fix nepokryje komplexní logiku s JOINy přes více unreleased tabulek
  • Vždy review quick fix výsledku — nestačí „apply all"
  • 10–20% případů vyžaduje ruční redesign datové logiky

⚠️ Joule pro legacy kód je omezený

Joule (GitHub Copilot pro ABAP) je trénovaný na SAP standardu, ne na zákaznickém Z-kódu. Neočekávej, že Joule automaticky přepíše 30 let legacy kódu. Je nejsilnější při:

  • Psaní nového Clean Core kódu (generování RAP artefaktů)
  • Unit testů pro existující kód
  • Vysvětlení co SAP released API dělá

⚠️ Příliš ambiciózní cíle — varování z praxe (Patrick Weber)

„Cíle byly příliš ambiciózní. Pro část User-Exitů released náhradník prostě neexistuje.

ATC job na 30 let legacy → 2,4 milionu errorů."

Klíčový takeaway: každý krok správným směrem se počítá. Clean Core není sprint — je to maraton. Zákazníkovi je třeba nastavit realistická očekávání.

⚠️ SmartForms / SAPScript → Adobe Forms — chybí týmová expertíza

Formulářová migrace je technicky jednoduchá v teorii, ale v praxi náročná na specialistu:

  • Tým ABAP vývojářů zpravidla neumí Adobe LiveCycle Designer (SFP transakce)
  • ADS na BTP vyžaduje samostatnou konfiguraci a licenci — není součástí S/4HANA on-premise licence
  • Weber's projekt tuto oblast označil jako jednu z nejtěžších — tým musel přizvat externího specialistu
  • Doporučení: plánovat formulářovou migraci jako separátní workstream s dedikovaným zdrojem

⚠️ Background joby — kaskádové závislosti na unreleased tabulkách

Migrace background jobu na Application Job Framework neznamená jen přepsat JOB_OPEN/CLOSE:

  • Většina legacy jobů čte přímo z unreleased tabulek (BKPF, MARA, EKKO...) — migrace jobu = migrace celé datové logiky
  • Joby mohou záviset na starých FM nebo FORM rutinách — každá závislost je další migrační krok
  • SAP Application Job Framework vyžaduje třídu implementující IF_APJ_RT_RUN — nelze zavolat existující Z-program přímo
  • Doporučení: background joby migrovat vždy v rámci většího datového refaktoringu, ne izolovaně

⚠️ Partnerské systémy nejsou připraveny na REST/OData

Migrace z RFC na OData API naráží na protistranu:

  • Starý ERP partner (jiný SAP nebo non-SAP) mluví RFC/IDoc — nelze ho přepnout jednostranně
  • Interní middleware (XI/PI/PO) může být nakonfigurovaný výhradně pro RFC adaptéry
  • Tier-2 RFC proxy (SAP-samples/tier2-rfc-proxy) řeší volající stranu, ale ne volaný legacy systém
  • Doporučení: integrační migrace vyžaduje koordinaci s IT architektem a mapování všech A2A/B2B toků ještě před zahájením

⚠️ BTP licencování — nečekané náklady

Část Clean Core vzorů přesouvá funkcionalitu na BTP — ale BTP není zdarma:

  • Adobe Document Services (ADS) = samostatná BTP service s consumption-based cenou
  • SAP Build Process Automation (workflow náhrada) = licence per workflow instance nebo user
  • BTP ABAP Environment (pokud zákazník chce cloud extensibility) = compute jednotky
  • Integration Suite (pro REST orchestraci) = integration flows licence
  • Doporučení: před architektonickými rozhodnutími vypočítat TCO včetně BTP nákladů — sticker shock může projekt zastavit

⚠️ HR a Payroll — oblasti s extrémně malým počtem released API

HR/HCM modul má historicky nejnižší pokrytí released API v celém SAP portfoliu:

  • Přímý přístup na infotype tabulky (PA0001, PA0002...) = Level D, ale released náhrada chybí pro velkou část scénářů
  • SAP HCM na S/4HANA je v maintenance mode — migrace na SuccessFactors je doporučená cesta
  • Zákazníci kteří nemigrují na SF musí akceptovat Level C pro mnoho HR scénářů
  • Doporučení: HR moduly inventarizovat zvlášť, konzultovat s SAP HCM specialistou před commitmentem na timeline

⚠️ Implicit enhancements — neviditelné v standardním Code Inspector

Implicitní enhancementy (ENHANCEMENT ... ENDENHANCEMENT) jsou nejrizikovější forma rozšíření:

  • Při prohlížení SAP standardního kódu (SE80) je vývojář nevidí — jsou skryté v enhancement sectionech
  • Při upgradu SAP standardu může SAP přidat kód na stejné místo nebo přejmenovat proměnné → silent break
  • SCMON nesleduje volání — nemáme UPL data pro prioritizaci
  • KernSeife kód EF9 score 10 Level D — nejvyšší rizikové skóre ze všech kategorií
  • Doporučení: spustit speciální report pro vyhledání implicit enhancements (SM84 + enhancement spot report) jako první krok inventury

⚠️ Unit testy neexistují — refaktoring bez záchytné sítě

Legacy kód má zpravidla nulové nebo minimální unit test pokrytí:

  • Přepsat Z-FM nebo Z-program bez unit testů = nemáme žádnou garanci zachování funkcionality
  • Tlak na „oprav to rychle" vede k: přepis kódu bez testů → bug v produkci → zákazník ztrácí důvěru v Clean Core projekt
  • Doporučení: nejdříve napsat unit testy pro existující logiku (Weber's approach), teprve potom refaktorovat. Testy dokumentují co kód dělá i když vývojář odejde.

⚠️ KernSeife score 0 neznamená automaticky „v pořádku"

Zdánlivě paradoxní: MIG (Migrace nutná) a NOC (Chybí klasifikace) mají score 0, ale Level D:

  • MIG = SAP explicitně říká, že objekt musí být migrován — score 0 neznamená OK, znamená „toto se nepočítá do váhy, ale MUSÍ být řešeno"
  • NOC = objekt není v klasifikačním souboru — může jít o nový objekt, nebo o chybu v klasifikaci; vyžaduje manuální review
  • Doporučení: sledovat MIG a NOC findings jako samostatnou kategorii — Level D objekty s nulovým score jsou v reportech „neviditelné" pokud reportujete jen sumární score

Doporučení: Spusťte KernSeife pro automatizované měření v ATC. Weberův index používejte jako komunikační formát pro management — je srozumitelnější pro netechnické stakeholders.

⚠️ Kontext pro management: KernSeife skóre měří architektonický soulad se SAP Cloud směrem, nikoliv funkčnost ani business hodnotu kódu. Snižování skóre je dobrý cíl pokud je řízeno byznysovými prioritami (cloud migrace, AI adopce, zjednodušení upgradů). Samotné snižování skóre bez kontextu není cíl — je to vedlejší efekt dobrých architektonických rozhodnutí. Viz také sekce 5a — Kde Clean Core nedává smysl.
