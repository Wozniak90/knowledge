💡 Hlavní myšlenka

Strukturovaný přístup k code review pomocí akronymu PERFECT snižuje kognitivní zátěž recenzentů, odstraňuje subjektivitu a dělá z review rutinní (ne vyčerpávající) proces — doplněný AI nástroji a měřitelnými metrikami.

Proč code review?

  • Může zabrat 20–30 % vývojářova času → vyplatí se jen strukturovaně
  • Cíle (dle Swarmia): sdílení znalostí, sdílená ownership, unifikace stylu, quality control
  • Pokud review existuje jen formálně (rubber stamp „LGTM") → lepší je self-review
  • Pokud odhaluje komunikační problémy → řeš příčinu, neruš review

PERFECT — prioritizovaný seznam principů

#PísmenoPrincipCo kontrolovat
1PPurposeŘeší kód zadaný úkol? Rozumím problému i řešení?
2EEdge CasesBoundary values, nullability, „impossible cases", neočekávané vstupy
3RReliabilityPerf (O-složitost), security, validace vstupů/výstupů, cache invalidation
4FFormSOLID, DRY, KISS → ale hlavně: High Cohesion / Low Coupling
5EEvidenceTesty + CI pipeline musí projít; testy review stejnými principy jako prod kód
6CClarityJmenné konvence, délka bloků, čitelnost bez čtení každého řádku
7TTasteOsobní preference — zaznamenat, ale neblokovat merge
⚠️ Pravidlo: Pořadí = priorita. Kód bez Purpose nemá hodnotu bez ohledu na čistotu.

Praktická doporučení & PR workflow

  1. Psané konvence — definuj kritéria předem, aktualizuj po každém vzorci
  2. Self-review před peer review — šetří čas všem, zlepšuje schopnosti autora
  3. PR workflow: DRAFT → READY FOR REVIEW → CHANGES REQUESTED → APPROVED → MERGED
    • Diskutuj high-level přístup před implementací (draft PR jako PoC)
    • Optimalizuj pro rychlost týmu, ne jednotlivce (Google Engineering Practices)
  4. Async vs. sync review — distribuované týmy preferují async; pair review pro komplexní změny
  5. Automatizuj — linting, CI, formatování → neplýtvej review čas na stroje
  6. Zastav „LGTM" — signál, že reviewer nepochopil změny
  7. Deliberate practice — review je skill, zlepšuje se tréninkem

Conventional Comments

Strukturovaný formát komentářů snižuje nejasnosti a tón diskuse:

<label> [decorations]: <subject>
[discussion]

Labely: suggestion, issue, question, nitpick, thought, chore, praise
Dekorace: (non-blocking), (blocking), (ux), (security)

Příklad:

issue (non-blocking): Tato metoda postrádá null check.
Pokud bude volána z nového endpointu, může spadnout. Navrhuju přidat early return.

🤖 AI-asistovaný code review (2026)

AI nástroje mění způsob, jakým code review funguje — doplňují, ne nahrazují peer review:

NástrojPopisIntegrace
GitHub Copilot Code ReviewInline AI review přímo v PR na GitHubu; navrhuje opravyGitHub
CodeRabbitNejrozšířenější AI review bot (3M repozitářů, 75M nalezených defektů); summary diffu + architekturální diagramGitHub, GitLab
Copilot / LLM jako self-reviewAutor může před odesláním PR nechat AI projít kód → redukce triviálních chybIDE
Doporučený přístup:
AI pokrývá PERFECT principy 3–6 (Reliability, Form, Evidence, Clarity).
Lidský reviewer se zaměří na Purpose a Edge Cases — kde AI selhává.

🏗️ SAP/ABAP kontext — doplňující perspektiva

Původní PERFECT framework je psán pro SaaS/product development. V enterprise SAP projektech platí navíc:

PERFECT↔ABAP mapování

PERFECTABAP specifický fokus
PurposeŘeší změna business případ? Je nutná modifikace standardu, nebo existuje BAdI/enhancement spot?
Edge CasesBoundary values v SAP typech (DATE, NUMC, CURR); chování při prázdné tabulce; volání z batch job vs. dialog
ReliabilityAuthority checks, SELECT performance, lock objects (ENQUEUE/DEQUEUE), SM12 deadlocks
FormClean ABAP Style Guide; OOP vs. procedural; obsolete konstrukce (FORM, MOVE, WRITE)
EvidenceABAP Unit Tests + ATC (ABAP Test Cockpit) green; transport testovaný na QA
ClarityNaming conventions (Z/Y prefix, snake_case v OOP); komentáře v EN nebo DE dle konvence týmu
TastePřebytečné konvence z doby před ABAP 7.40 — neblokovat merge, zaznamenat do team guidelines

Rozšířený checklist pro ABAP review

Reliability & Security

  • SELECT * → nahradit explicitním výběrem polí (výkon + přehlednost)
  • Authority checks — AUTHORITY-CHECK přítomno u všech citlivých READ/CHANGE operací?
  • SQL injekce — dynamické WHERE podmínky ošetřeny CL_ABAP_DYN_PRG?
  • Lock objects — souběžný přístup ošetřen ENQUEUE před změnou dat?

Clean Core & Upgrade

  • Modifikace standardu — je to nutné? Existuje extension point (BAdI, Customer Exit, Enhancement Spot)?
  • Backward compatibility — poruší změna existující interface nebo BAPI signaturu?
  • Deprecated API — používá kód CALL FUNCTION místo OOP alternativy?
  • Namespace — objekty ve správném Z/Y namespace?

Transport & Process

  • Transport request — správná kategorie (Workbench vs. Customizing), správný systém/klient?
  • Žádné hardcoded hodnoty závislé na systému (client ID, hostname, RFC destination)?

Automatizace review v ABAP světě

NástrojPopisEkvivalent
ABAP Test Cockpit (ATC)SAP built-in statická analýza; security, performance, syntax; integrovaný v ADT/EclipseGitHub Actions

abapLintOtevřený linter spustitelný v CI/CD (GitHub Actions) — funguje na exportovaném kódu přes abapGitESLint/RuboCop
ABAP Unit TestsVestavěné unit testy; povinné v Clean Core / ABAP Cloud; spustitelné z ATCJest / RSpec
💡 Doporučení: ATC + Code Pal for ABAP pokrývá PERFECT principy 3–6. Lidský reviewer se soustředí na Purpose (business correctness) a Edge Cases (domain-specific).

Odlišné priority oproti SaaS

  • Compliance & audit trail — zákonné požadavky > estetika kódu (GDPR, SOX, GxP v pharma)
  • Backward compatibility — upgrade readiness, clean core princip (S/4HANA Cloud)
  • CI/CD omezení — ABAP Unit Test ≠ GitHub Actions; abapGit + abapLint umožňují částečnou automatizaci
  • Transport landscape — DEV → QA → PROD je explicitní gate; review musí proběhnout před requestem

💭 Vlastní reflexe

  • PERFECT pyramida: Taste na konci brání bikesheddingu — v hierarchických vztazích (klient/vendor) dvojnásob důležité
  • „Impossible cases" — nevědomé předpoklady v kódu jsou podceňovaná kategorie chyb
  • AI review (CodeRabbit, Copilot) pokrývá mechanické aspekty → uvolňuje lidi pro Purpose & Edge Cases
  • V ABAP světě je F (Form) klíčový: Clean ABAP Style Guide + DSAG leitfaden = základ konvencí
  • Conventional Comments výrazně zlepšují tone of review, zejména v multi-kulturních týmech
