Code Review — PERFECT Framework

💡 Hlavní myšlenka (1 věta)

Strukturovaný přístup k code review pomocí akronymu PERFECT snižuje kognitivní zátěž recenzentů, odstraňuje subjektivitu a dělá z review rutinní (ne vyčerpávající) proces — doplněný AI nástroji a měřitelnými metrikami.

Obsah

Proč code review?

  • Může zabrat 20–30 % vývojářova času → vyplatí se jen strukturovaně
  • Cíle (dle Swarmia): sdílení znalostí, sdílená ownership, unifikace stylu, quality control
  • Pokud review existuje jen formálně (rubber stamp "LGTM") → lepší je self-review
  • Pokud odhaluje komunikační problémy → řeš příčinu, neruš review

PERFECT — prioritizovaný seznam principů

# Písmeno Princip Co kontrolovat
1 P Purpose Řeší kód zadaný úkol? Rozumím problému i řešení?
2 E Edge Cases Boundary values, nullability, "impossible cases", neočekávané vstupy
3 R Reliability Perf (O-složitost), security, validace vstupů/výstupů, cache invalidation
4 F Form SOLID, DRY, KISS → ale hlavně: High Cohesion / Low Coupling
5 E Evidence Testy + CI pipeline musí projít; testy review stejnými principy jako prod kód
6 C Clarity Jmenné konvence, délka bloků, čitelnost bez čtení každého řádku
7 T Taste Osobní preference — zaznamenat, ale neblokovat merge

⚠️ Pravidlo: Pořadí = priorita. Kód bez Purpose nemá hodnotu bez ohledu na čistotu.

Praktická doporučení & PR workflow

  1. Psané konvence — definuj kritéria předem, aktualizuj je po každém vzorci (comment → diskuse → fix)
  2. Self-review před peer review — šetří čas všem, zlepšuje schopnosti autora
  3. PR workflow: DRAFT → READY FOR REVIEW → CHANGES REQUESTED → APPROVED → MERGED
  4. Diskutuj high-level přístup před implementací (draft PR jako PoC)
  5. Optimalizuj pro rychlost týmu, ne jednotlivce (Google Engineering Practices)
  6. Async vs. sync review — distribuované týmy preferují async (PR komentáře); pair review je alternativa pro komplexní změny
  7. Automatizuj — linting, CI, formatování → neplýtvej review čas na stroje; viz sekce AI níže
  8. Zastav "LGTM" — signál, že reviewer nepochopil změny; snižuje hodnotu procesu
  9. Deliberate practice — review je skill, zlepšuje se tréninkem

Conventional Comments

Strukturovaný formát komentářů snižuje nejasnosti a tón diskuse:

<label> [decorations]: <subject>
[discussion]

Labely: suggestion, issue, question, nitpick, thought, chore, praise
Dekorace: (non-blocking), (blocking), (ux), (security)

Příklad:

issue (non-blocking): Tato metoda postrádá null check.
Pokud bude volána z nového endpointu, může spadnout. Navrhuju přidat early return.

🤖 AI-asistovaný code review (2026)

AI nástroje mění způsob, jakým code review funguje — doplňují, ne nahrazují peer review:

Nástroj Popis Integrace
GitHub Copilot Code Review Inline AI review přímo v PR na GitHubu; navrhuje opravy GitHub
CodeRabbit Nejrozšířenější AI review bot (3M repozitářů, 75M nalezených defektů); summary diffu + architekturální diagram GitHub, GitLab
Copilot / LLM jako self-review Autor může před odesláním PR nechat AI projít kód → redukce triviálních chyb IDE

Doporučený přístup: - AI pokrývá PERFECT principy 3–6 (Reliability, Form, Evidence, Clarity) — ty rutinní - Lidský reviewer se zaměří na Purpose a Edge Cases — kde AI selhává - AI review nenahrazuje business porozumění ani domain knowledge

📊 Metriky code review

Bez měření se proces zlepšuje jen náhodně. Klíčové metriky:

Metrika Co měří Zdravá hodnota
PR cycle time Doba od otevření PR po merge < 1 pracovní den (malé PR)
Review throughput Počet PR reviewovaných za sprint závisí na týmu
Defect escape rate % bugů nalezených až v produkci (ne v review) klesající trend
Review coverage % PR s alespoň 1 reálným review komentářem > 80 %
Time to first review Jak rychle dostane PR první feedback < 4 hodiny

🏗️ SAP/ABAP kontext — doplňující perspektiva

Původní PERFECT framework je psán pro SaaS/product development. V enterprise SAP projektech platí navíc:

Rozšířený checklist pro ABAP review: - [ ] SELECT * → nahradit explicitním výběrem polí - [ ] Authority checks — přítomny u všech citlivých operací? - [ ] Modifikace standardu — je to nutné? Je to clean core compatible? - [ ] Backward compatibility — poruší změna existující interface? - [ ] Transport request — správná kategorie, správný systém?

Odlišné priority oproti SaaS: - Compliance & audit trail — zákonné požadavky > estetika kódu - Backward compatibility — upgrade readiness, clean core princip - CI/CD omezení — ABAP Unit Test ≠ GitHub Actions; pipeline je složitější nastavit

💭 Vlastní reflexe

  • PERFECT pyramida: Taste na konci brání bikesheddingu — v hierarchických vztazích (klient/vendor) je to dvojnásob důležité
  • "Impossible cases" — nevědomé předpoklady v kódu jsou podceňovaná kategorie chyb
  • AI review (CodeRabbit, Copilot) pokrývá mechanické aspekty → uvolňuje lidi pro Purpose & Edge Cases
  • V ABAP světě je F (Form) klíčový: Clean ABAP Style Guide + DSAG leitfaden = základ konvencí
  • Conventional Comments výrazně zlepšují tone of review, zejména v multi-kulturních týmech
