1) LLM základy

  • LLM = transformery trénované na desítkách trilionů tokenů (healthcare, law, coding, science…)
  • Populární modely: OpenAI GPT, Anthropic Claude, Google Gemini
  • Token ≈ ¾ anglického slova

  • 1M tokenů ≈ 7,5k slov nebo 50k řádků kódu
  • Klíčový poznatek: 500 GB firemních dokumentů se do kontext window prostě nevejde → potřeba embeddingů + RAG
  • Nerelevantní kontext může LLM zmást (příklad s počítáním jablek s nesouvisejícími detaily)

2) Embeddingy a sémantická reprezentace

  • Text → numerický vektor ve vysokodimenzionálním prostoru (typicky 1536 dimenzí)
  • Sémanticky podobné texty = matematicky blízké vektory
  • Příklad: "employee vacation policy" ≈ "staff time off guidelines"
  • Příklad z praxe: dotaz "I wear jeans to work" vrátí dress code policy, i když slovo „jeans" v dokumentu není

3) LangChain – orchestrační vrstva

Proč? Přímé SDK vyžaduje hodně boilerplatu – LangChain to zjednodušuje:

  • Standardizovaný interface pro různé providery (OpenAI → Anthropic = změna jednoho řádku)
  • Komponenty: chat modely, Memory Saver (chat history), vector DB interface (Pinecone, ChromaDB), embedding utils, tools (DB, web search, file system)

LLM vs. Agent:

  • LLM = statický mozek odpovídající z tréninkových dat
  • Agent = autonomní systém s pamětí, nástroji a schopností multi-step rozhodování

LangChain v kódu:

  • ~70% méně kódu než raw OpenAI SDK
  • Prompt templates (placeholders místo hard-coded promptů)
  • Output parsers (převod textu na list/JSON)
  • Chain kompozice: prompt → model → parser (pipe-like)

5) Vector Databases a sémantické vyhledávání

Problém s klasickým vyhledáváním: keyword/SQL search selže u sémanticky příbuzných dotazů ("reset password" vs "password recovery").

Jak to funguje:

  1. Dokument se rozseká na chunky (s překryvem pro zachování kontextu)
  2. Každý chunk → embedding model → vektor
  3. Vektory se uloží do vector DB (Pinecone, ChromaDB)
  4. Dotaz se také převede na vektor → similarity search

Klíčové parametry:

  • Dimenzionalita (typicky 1536) – vyšší = bohatší kontext, ale větší velikost
  • Similarity threshold – nastavit, aby se filtrovaly slabé shody
  • Chunk size + overlap – overlap zachovává kontext přes hranice chunků

6) RAG (Retrieval Augmented Generation)

Kombinace vector DB search + LLM:
Dotaz → embedding → podobné chunky → LLM dostane relevantní kontext → odpověď

Praktický příklad: TechCorp's 500 GB dokumentů prohledáno za < 30 sekund.
